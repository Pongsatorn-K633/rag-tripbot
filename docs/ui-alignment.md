# UI & Motion Alignment Guide (SSOT)

> **Purpose:** the single source of truth for how UI and motion are built in this app, so that
> anything generated by an external tool (e.g. **Kimi**) or a fresh session **matches the existing
> codebase** instead of drifting to new libraries, off-palette colors, or dead patterns.
>
> **How to use with an external tool:** feed it **this file** (the *rules*) +
> **[`docs/landing-frontend-dump.md`](landing-frontend-dump.md)** (the *current code* — landing
> page, navbar, shell, `globals.css` tokens, `IMG`, regenerated from source). Together they are the
> full foundation. Then paste the generated code back here for integration + `/code-review`.
>
> CLAUDE.md → "UI / Design Conventions" is the short quick-ref; **this file is the detailed SSOT.**

---

## 0. How to read this — the user's prompt is the goal

**This guide is guardrails, not a spec to reproduce.** The user's request / target design is what
you're building; this doc only keeps the *result* consistent with the app. **If a rule here would
block the change the user asked for, follow the user's intent and note the trade-off — do not
shrink, skip, or water down the change to satisfy a convention.** You are expected to make
substantial changes when asked (add sections, restyle, new motion); the code dump shows what
exists so your output *fits in*, not so you copy it.

Only the short list below is non-negotiable. **Everything in §1–§7 beyond it is a *default* —
follow it when it doesn't fight the requested design.**

**Hard invariants (don't break these):**
1. **Single cool palette — no dark mode**, no `dark:` variants. Keep *brand* colors on palette tokens.
2. The hero → `#0A1B33` bottom-fade **seam stays seamless** (§3).
3. Keep `id="pathways"` as the scroll target; keep interactive elements clickable; honor `prefers-reduced-motion`.
4. Don't animate `scale` on the logo/photo **bitmaps** (it pixelates) — animate transform/layout instead.

Beyond those four, treat the rest as strong suggestions you can override for a better result.

## 0.1 How to prompt this tool (Kimi) for a change

The framing of the prompt matters more than the rules. To get a *real* change instead of a
near-copy of the current UI:

1. **Lead with the change; reference this doc second.** e.g. *"Redesign the `#pathways` section
   into an animated parallax showcase — [intent]. Keep it consistent with this guide (guardrails
   only — the 4 hard invariants in §0); change freely otherwise."*
2. **Don't paste the whole code dump for a targeted change.** Feed **this doc + only the file
   you're changing** (e.g. `app/page.tsx`). The full shell biases the tool toward "preserve
   everything." Use the complete dump only when it truly needs cross-file context.
3. **Ask for the changed section, not the whole file** — *"return just the new `#pathways`
   section"* — so it isn't anchored to reproducing hundreds of lines.
4. **For visual changes, give a target image.** A mockup / reference screenshot drives the look
   far more than the existing code does.
5. **Name what may change.** *"You may replace the content grid, add sections, restyle the button —
   only the hero seam and palette are fixed."* Explicit permission beats implied.

## 1. Design tokens

Values live in `app/globals.css` `@theme` (that file is the value SSOT). **Prefer** Tailwind
tokens for *brand* colors so a palette change propagates app-wide; one-off effect colors as raw
hex are fine (like the hero gradient stops in §3). Just don't hardcode a brand color that should
be a token.

**Palette — cool, single scheme, NO dark mode:**
- `zen-black` `#122C4F` — Midnight: dark base + body text
- `briefing-cream` `#F7F9FC` — Cloud: light background + text-on-dark
- `basel-brick` `#5B88B2` — Ocean: **the single accent** (buttons / links / hover / active). **No red anywhere.**
- `noir` `#000000` — true black, for deep-contrast moments
- `graphite` `#334155` — cool dark grey (Midnight's hue, desaturated): text on light glass/milk surfaces
- `hinomaru` `#D80027` — Japan-flag red. **Content color only** (the flag badge) — never a UI accent; the no-red rule stands

**No theme switching.** Light is the only scheme. Do **not** add `dark:` variants or hardcoded
`bg-white`. There is no `ThemeProvider` — dark mode was removed entirely (2026-07).

**Fonts:** `font-headline` (Manrope) for display/UI · Inter for body · Noto Sans Thai for Thai.
Thai headers scale responsively: `text-3xl md:text-5xl`.

## 2. Motion rules

- **Library:** default to `motion/react` (Framer Motion, already installed) for entrance/scroll
  motion, and CSS transitions for hover/scroll state. Prefer not to pull in another animation lib
  (`react-scroll-parallax`, GSAP, AOS…) — but if an effect genuinely needs one, use it and say why.
- **Scroll-linked motion:** `useScroll()` + `useTransform()`, applied as `translateY` / `y` (or
  opacity) on layers.
- **No animated `scale` on bitmaps.** Scaling an image or a rounded element while it animates
  pixelates the bitmap and distorts border-radii. Animate real layout props instead
  (`width`/`height`, `translate`, `grid-template-columns`). A *static* baseline scale (e.g. a
  fixed `scale-110` that never animates) is fine.
- **`prefers-reduced-motion`:** gate every non-trivial motion behind `useReducedMotion()` (or the
  CSS media query) and fall back to a static layout.
- **Decorative layers** are `pointer-events-none` so interactive elements underneath stay clickable.
- **Responsive sizing:** fluid `clamp(min, vw, max)` for hero type/padding; a `sizes` prop on
  every `next/image`.

## 3. Landing hero anatomy (`app/page.tsx`)

The hero is a `h-screen min-h-[660px] overflow-hidden` section with stacked layers. **Keep this
z-order** when adding motion:

| z | Layer | Notes |
|---|---|---|
| `z-0` | Full-bleed `next/image` (`object-cover object-[center_42%]`) | the photo |
| `z-10` | Midnight legibility gradient (top→bottom) | keeps text readable |
| `z-10` | **Bottom fade** → ends on **exactly `#0A1B33`** | see the seam rule below |
| `z-15` | JAPAN wordmark + Start Journey button | centered content |
| `z-20` | Learn More cue + socials | bottom corners |

*(The navbar is a separate `fixed z-50` element — not part of the hero stack.)*

**The seam rule (critical):** the section below the hero starts with
`linear-gradient(180deg,#0A1B33 0%,#122C4F 100%)`. The hero's bottom-fade layer dissolves the
photo into **exactly `#0A1B33`** so there's no visible line between them. `#0A1B33` and `#122C4F`
are the **only** raw hex allowed. Any motion that moves the photo (e.g. parallax translating it
upward) **must keep the bottom-fade layer pinned to the section bottom**, or the hard seam
reappears.

*Content layer note:* the JAPAN wordmark and Start Journey button are nudged up slightly
(`translateY(-10%)` / `-22%`) for composition — a static baseline offset, not animated.

### Page structure below the hero (as of 2026-07-06)

After the hero, the lower page shares **one continuous Midnight gradient**
(`linear-gradient(180deg,#0A1B33 0%,#122C4F 100%)`) wrapping two sections:

1. **`#pathways`** — currently a **reserved empty band** (`min-h-[860px]`, `px-8 py-24`,
   `text-briefing-cream`). The old 3 pathway cards were **removed**; **this is where the new
   landing section / scroll-motion goes.** Keep `id="pathways"` (the hero's Start Journey + Learn
   More scroll to it) and the padding.
2. **Content-preview grid** — a 4-cell image mosaic (Tokyo Nights / Zen Heritage / The Peak + a
   `basel-brick` "100K+ Explorers" tile), grayscale → color on hover.

## 4. Parallax guardrails (landing hero)

When adding scroll parallax to the hero, on top of §2:

1. **`motion/react` only** — `useScroll()` + `useTransform()` → `translateY` on layers. No new lib.
2. **Preserve the z-layer stack** in §3. Move layers at *different rates* but keep their order.
3. **Don't break the seam.** If the bg image translates upward, keep the bottom-fade layer pinned
   to the section bottom (still ending on `#0A1B33`) — see §3.
4. **No animated `scale`** on the photo. To avoid revealing edges as it moves, give the image
   container a *static* larger size (e.g. `h-[120%]` or a baseline `scale-110`) — fixed, never
   animated.
5. **`prefers-reduced-motion`** → disable/attenuate the parallax (`useReducedMotion()`).
6. **Decorative layers stay `pointer-events-none`** so Start Journey / socials stay clickable.
7. **Palette/tokens** — any new gradient/overlay uses `zen-black` / `briefing-cream` /
   `basel-brick`; the only raw hex allowed are the existing `#0A1B33` / `#122C4F` stops.

## 5. Navbar motion vocabulary (`app/components/Navbar.tsx`)

Reference implementation for the app's signature motion. Reuse this vocabulary for consistency:

- **Transparent over the hero** at the top; on scroll the pieces animate in.
- **Center nav oval:** a `briefing-cream` oval blooms from the middle — `width 0→100%` +
  `height 0→46px`, centered via `-translate-x/y-1/2` — to wrap the nav items.
- **Side pills** (left logo + "dopamichi", right user menu): unfurl **left→right** — anchored
  `left-0`, `width 0→100%` + `height 0→46px` — so they grow out of the leading circle. Style:
  `bg-black/20`, `group-hover:bg-black/30`, `transition-[width,height] duration-[700ms] ease-in-out`.
- All three share a **46px height** and a **34px leading circle** for a consistent family.
- Content sits on top of the blooming background via `relative` (positioned, later in DOM).

## 6. Integration review (post-hoc QA — NOT a design constraint)

This is a checklist the **human integrator** runs *after* the UI is generated — a quality pass, not
a limit on what may be designed. A "no" here is a fix-up, not a reason to reject the new design.
Before merging:

- [ ] No new animation/UI libraries added (uses `motion/react` + Tailwind + `next/image`).
- [ ] Colors are palette tokens, not raw hex (except the two hero gradient stops).
- [ ] No `dark:` variants or hardcoded `bg-white`.
- [ ] No animated `scale` on images/rounded elements.
- [ ] `prefers-reduced-motion` handled.
- [ ] Hero z-order + bottom-fade seam intact (`#0A1B33`).
- [ ] Interactive elements not covered by `pointer-events` layers.
- [ ] `next/image` has a `sizes` prop; fluid `clamp()` sizing where relevant.
- [ ] Fonts via `font-headline` / Inter / Noto Sans Thai; Thai headers `text-3xl md:text-5xl`.
- [ ] Prose/doc links are markdown `[file](path)`, not backticks.

## 7. Brand assets — logo & favicons (two SEPARATE sources)

Updating one does **not** update the other — a mascot change must touch both:

- **Navbar logo:** `public/android-chrome-192x192.png` (+ `-512x512.png`), used in
  `app/components/Navbar.tsx` via `IMG.logo`.
- **Browser-tab favicons:** `public/favicon.ico`, `favicon-16x16.png`, `favicon-32x32.png`,
  `apple-touch-icon.png` — wired in `app/layout.tsx` `metadata.icons`. **Filenames are fixed**
  (referenced by `layout.tsx`); replace their *contents*, never rename them.

**When the mascot changes, regenerate the favicons from the 512 master** (they're not
auto-derived): resize `android-chrome-512x512.png` → 16/32 PNGs + a 180 `apple-touch-icon.png`,
and assemble a multi-size `favicon.ico` (via `sharp`, already a dependency). Browsers cache
favicons aggressively — hard-refresh (Ctrl+Shift+R) / bust the CDN cache to see the change.

*History — 2026-07-05:* the favicons had lagged behind the navbar logo (still the old magnifier)
and were regenerated (contents only, no renames) from the current shinkansen-bird mascot.
